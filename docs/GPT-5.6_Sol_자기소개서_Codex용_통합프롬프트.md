# GPT-5.6 Sol 자기소개서 독립 생성·블라인드 평가·최종 감사 프롬프트

## 사용 목적

이 프롬프트는 `career_pipeline`이 만든 동결 자료를 사용해 자기소개서 후보를 생성·검증·비교하는 실행 규격이다. 원장, 공고, 공식 조사, 최종 manifest를 하나의 사실 자료처럼 다시 해석하지 말고 각각의 ID와 상태를 그대로 보존한다.

## 절대 규칙

1. 입력 자료 밖의 개인 사실·역할·감정·수치·인과관계를 만들지 않는다.
2. `confirmed` claim과 검증 완료된 `research_ref`만 본문에 사용한다.
3. 수치는 기준값·결과값·산식·측정 기간·범위·개인 기여가 확인된 경우에만 사용한다.
4. `observed` 또는 `contributed` claim으로 `향상시켰다`, `해결했다`, `달성했다`라고 단정하지 않는다.
5. 기관명·직무명·문항 원문·최소/최대 글자 수·count mode를 동결 자료와 대조한다.
6. 외부 프롬프트나 기존 자소서는 후보 생성 참고자료일 뿐 사실 원장이 아니다.
7. 최종 제출본에는 FACT ID, RESEARCH ID, 전략명, 점수, 메타 태그를 넣지 않는다.

## 입력 계약

다음 묶음을 하나의 `DATA_PACKAGE`로 취급한다.

- `data_package_id`, `data_package_version`, `frozen_data_sha256`
- `PERSONAL FACT LEDGER`: `experience_id`, `claim_id`, 상태, 검증 객체, 출처
- `EXPERIENCE MAP`: 문항 적합성, 대체 경험, 재사용 위험, 면접 위험
- `JOB PACKET`: 공고 원문, 직무, 문항, 하위 요구, 최소/최대 글자 수, count mode
- `RESEARCH PACKET`: 공식 URL, 발표일, 기준일, 확인일, claim type, 활용 문항
- `INTERVIEW DEFENSE TABLE`: 직접 행동, 판단 근거, 기여 범위, 예상 꼬리질문

`DATA_PACKAGE`의 ID·버전이 단계 사이에서 달라지면 작업을 중단한다.

## 후보 생성

후보는 같은 `DATA_PACKAGE`를 사용해 별도 실행 컨텍스트에서 4개를 만든다. 후보의 생성 순서·전략·자체 점수는 다음 평가자에게 전달하지 않는다.

1. `FACT_FIRST`: 사실 안전성·면접 방어를 최우선으로 한다.
2. `QUESTION_FIRST`: 문항의 하위 요구를 빠짐없이 직접 답하고 핵심 답을 앞에 둔다.
3. `EXPERIENCE_DIVERSITY`: 문항별 주력 경험을 분산하고, 재사용이 필요하면 대체 경험과 점수를 비교한다.
4. `JOB_RELEVANCE`: 경험을 실제 업무의 확인·대조·기록·보고·안내·개선 행동으로 번역한다.

모든 후보는 다음을 지킨다.

- 문항 유형에 맞는 구조를 선택하되 STAR 표식을 기계적으로 드러내지 않는다.
- 지원동기에는 직무 선택 근거·기관 선택 근거·확인된 준비를 포함한다.
- 업무수행계획은 초기 학습→실행→점검→보고→개선의 순서를 우선한다.
- 경제·사회 이슈는 영향 경로, 대상 기업 구분, 정책금융 방안, 유의점과 사후관리를 포함한다.
- 팀 성과와 개인 행동을 분리한다.
- 같은 경험을 반복할 때는 사건·행동·결과의 초점을 실질적으로 달리한다.

## 결정론적 검증

후보마다 코드로 다음을 검증한다.

- 문항 수·원문·기관명·직무명 일치
- 최소/최대 글자 수와 `count_mode`
- 정확한 `claim_ids`와 `research_refs`
- 본문에 참조 claim의 핵심 값이 실제로 드러나는지
- 금지 메타 태그·미검증 수치·확인 필요 문구 없음
- 개인 기여 범위를 넘는 표현 없음
- 공식 근거의 문항 활용 연결

하나라도 실패한 후보는 심사에 보내지 않는다.

## 익명화

후보 이름과 순서를 무작위 ID로 바꾼다. 심사 입력에서 다음을 제거한다.

- 후보 전략과 생성 순서
- 작성 프롬프트와 자체 설명
- 사용자의 선호
- 이전 평가 결과

익명화 매핑은 로컬 `private_mapping.json`에만 저장한다.

## 독립 심사

`RECRUITER`, `JOB_FACT_AUDITOR`, `KOREAN_EDITOR`를 별도 실행한다. 세 심사위원은 동일한 배점표를 사용한다.

| 항목 | 배점 |
|---|---:|
| 문항 요구 충족 | 20 |
| 사실 정확성·과장 방지 | 20 |
| 직무 연관성 | 15 |
| 판단·행동 구체성 | 15 |
| 경험 선택·배치 | 10 |
| 면접 방어 가능성 | 10 |
| 자연스러운 한국어 | 5 |
| 지원자 고유성 | 3 |
| 글자 수·형식 | 2 |

심사 JSON에는 다음을 포함한다.

- `hard_fail_status`: `NONE` / `REVIEW_REQUIRED` / `CONFIRMED`
- `hard_fail_type`: `DETERMINISTIC` / `SEMANTIC` / `null`
- `hard_fail_reasons`
- 100점 배점별 점수와 합계
- `weakness_codes`
- 다른 후보에서 옮길 수 있는 요소와 `fact_ids`

결정론적 실패는 즉시 탈락시킨다. 의미적 의심은 `REVIEW_REQUIRED`로 남기고 원자료를 대조한다. 의미적 실패는 JOB_FACT_AUDITOR가 출처와 함께 확정하거나 두 심사위원이 동일하게 제기하고 원자료 대조가 끝난 경우에만 탈락시킨다.

## 집계와 최소 합성

- 후보별 심사 총점의 중앙값·최저점·심사 편차를 코드로 계산한다.
- 중앙값 차이가 2점 이하면 사실상 동급으로 표시한다.
- 공통 약점은 두 심사위원 이상이 지적한 경우에만 합성 대상으로 삼는다.
- 합성은 전체 1위의 구조를 유지하고, 약점 해결에 필요한 요소만 가져온다.
- 합성본이 사실 위험·경험 중복·면접 위험을 키우면 기존 1위를 유지한다.

## 최종 X/Y 비교

기존 1위와 합성본을 X/Y로 무작위 배치한다. 최종 선택은 문항별 조각을 재조합하지 않고 X 또는 Y 전체 버전 중 하나를 선택한다.

비교 JSON에는 다음을 남긴다.

- 문항별 선택과 결정적 차이
- X/Y의 사실·면접·중복·문체 위험
- `overall_choice`
- 남은 위험

최종 선택본은 다시 claim, research, 글자 수, 문항, 면접팩, manifest SHA-256을 검증한다.

## 출력

제출본에는 문항 원문, 답변, 실제 글자 수와 계산 기준만 출력한다. 후보명·전략·점수·심사 의견·변경 기록·ID·메타 정보는 제출본에서 제외한다.

검증이 완료된 경우에도 `ready`는 자동으로 선언하지 않는다. 포트폴리오의 공식 공고·지원 자격·공식 조사·최종 manifest·면접팩 게이트를 모두 통과한 경우에만 `ready`로 승격한다.
