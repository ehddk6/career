# 지원 자동화 후속 설계

이 문서는 Phase 1 이후의 공고 탐색·자격 판정·지원서 입력·제한적 제출을 설계한다. 이번 Phase 1에서는 브라우저 자동화, 실제 로그인, 개인정보 전송, 지원서 입력과 제출을 구현하지 않는다.

## 1. 공식 공고 탐색

사용자가 지정한 공식 채용 사이트와 허용 도메인만 탐색한다. `PostingRecord`는 `posting_id`, `url`, `official_domain`, `published_at`, `deadline_at`, `title`, `organization`, `role`, `body_sha256`, `retrieved_at`, `source_status`를 가진다. HTTPS를 기본으로 하고 리다이렉트·응답 크기·도메인 정책은 기존 `posting_loader`와 같은 제한을 사용한다.

## 2. 공고 중복·변경 감지

공고 ID를 1차 키로 사용하고 ID가 없으면 정규화된 URL과 기관·직무 조합을 보조 키로 사용한다. 본문 SHA-256이 바뀌면 `changed` 이벤트를 기록하고 이전 본문과 새 본문을 보존한다. 마감일이나 문항이 바뀌면 기존 지원 패키지를 자동 제출 대상으로 재사용하지 않고 `manual_review`로 보낸다.

## 3. 지원자 프로필

`ApplicantProfile`은 승인된 경험 원장, 학력, 경력, 자격증, 근무 가능 지역, 개인정보 필드의 참조만 포함한다. `experience_ledger.json`의 `confirmed` claim만 자격·자소서·면접 근거로 사용하며 `proposed`, `stale`, `unknown`은 후보로만 남긴다.

## 4. 필수 자격 판정

공고의 필수 자격과 우대 조건을 분리해 `EligibilityDecision`으로 저장한다. 학력·경력·자격증·지역·지원 제한을 각각 `met`, `not_met`, `unknown`으로 판정하고 다음 상태로 집계한다.

```text
eligible
eligible_with_gaps
manual_review
ineligible
```

근거가 없거나 서로 충돌하거나 자연어 해석이 필요한 조건은 합격으로 추정하지 않고 `manual_review`로 보낸다. 우대 조건 미충족은 필수 조건 충족 여부와 분리한다.

## 5. 자기소개서 생성

`ApplicationTarget`이 공고 해시·자격 판정·문항 목록·승인 근거를 묶는다. 생성 결과는 기존 Phase 1의 `draft.json` 계약을 사용하고, 문항별 `experience_refs`, `research_refs`, `evidence_paths`, 글자 수, 검증 상태를 기록한다. 최종 선택은 `draft_final.json`과 `12_최종산출물.json`으로 고정한다.

## 6. 면접 claim graph

경험 claim을 노드로, 문항·공고 요구역량·면접 질문·근거 파일을 엣지로 연결한다. 수치·기간·역할·성과는 원장 claim 노드에서만 파생하며, 연결되지 않은 문장은 `unsupported_claim`으로 표시한다. 공고 변경 시 영향받는 문항과 면접 질문을 다시 계산한다.

## 7. 지원 패키지

`ApplicationPackage`는 개인정보 필드, 학력, 경력, 자격증, 자소서 문항·답변, 첨부파일, 지원 직무·지역, 공고 해시, 최종 자기소개서 해시, 생성 시각, 검증 결과를 포함한다. 모든 입력 파일은 해시와 상대 경로를 저장하고 원본은 수정하지 않는다.

## 8. Playwright 폼 어댑터

사이트별 `FormAdapter`가 필드 탐색, 값 입력, 파일 첨부, 화면 상태 캡처를 담당한다. 공통 계층은 CSS 선택자에 의존하지 않고 label/name/role 기반 매핑을 우선한다. 어댑터는 `discover`, `fill`, `verify`를 분리하고 제출 동작은 별도 메서드로 격리한다.

## 9. `review_required` 모드

초기 기본 모드는 `review_required`다. 공고가 확정되고 지원 패키지가 생성되어도 입력은 제출 직전에서 멈춘다. 사용자 검토 화면에는 공고 본문 해시, 각 필드 값, 첨부파일, 자소서 SHA-256, 자격 판정, 미확인 항목을 보여준다.

## 10. 제한적 자동 제출 조건

자동 제출은 다음 조건을 모두 만족할 때만 별도 명시적 설정으로 허용한다.

- 사용자가 해당 사이트의 자동 제출을 명시적으로 허용
- 허용 도메인 목록에 포함
- CAPTCHA·MFA 없음
- 준비 시점 이후 공고 본문·문항·마감일 불변
- 자격 판정이 `eligible`
- Phase 1 품질 게이트 통과
- 중복 지원 아님
- 모든 필드 입력 후 재검증 완료
- 접수번호 또는 완료 화면 저장 가능

조건 하나라도 빠지면 `review_required` 또는 `manual_review`로 되돌린다.

## 11. 중단 조건

CAPTCHA, MFA, 로그인 추가 확인, 새 문항, 알 수 없는 필드, 예상과 다른 파일 형식, 공고 해시 변경, 필수 필드 불일치가 나타나면 즉시 멈춘다. CAPTCHA·MFA 우회나 보안 질문 추측은 구현하지 않는다.

## 12. 중복 지원 방지

제출 전 `(site, organization, posting_id, applicant_identity_fingerprint)`와 공고 해시를 비교한다. 기존 접수 기록이 있거나 동일 공고의 진행 중 패키지가 있으면 자동 제출을 막고 사용자에게 병합·취소·재지원 결정을 요청한다.

## 13. 접수번호와 제출 증거

제출 완료 화면의 URL, 화면 캡처, 접수번호, 제출 시각, 공고 해시, 패키지 해시를 저장한다. 접수번호를 읽지 못하거나 완료 화면을 확인하지 못하면 성공으로 표시하지 않고 `submission_unverified`로 남긴다.

## 14. 개인정보 저장·암호화·로그 마스킹

주민등록번호·계좌·비밀번호·세션 쿠키는 일반 JSON이나 로그에 저장하지 않는다. 로컬 비밀 저장소 또는 운영체제 보호 저장소를 사용하고, 저장이 필요한 개인정보는 암호화된 별도 저장소에 둔다. 로그와 오류에는 이름·연락처·주소·식별번호·토큰을 마스킹하며, 외부 모델·검색어·사이트 폼으로 개인정보를 전송하지 않는다.

## 15. 사이트별 어댑터 테스트 전략

사이트별 HTML fixture와 mock browser를 사용해 다음을 검증한다.

1. 정상 필드 탐색과 label 매핑
2. 파일 첨부 및 글자 수 검증
3. 새 문항·알 수 없는 필드에서 안전한 중단
4. 공고 해시 변경에서 재검토 전환
5. CAPTCHA·MFA 감지와 우회하지 않는 동작
6. 중복 지원 차단
7. 완료 화면·접수번호·증거 저장 실패 처리

실제 채용 사이트를 대상으로 한 테스트는 승인된 테스트 계정과 제출하지 않는 sandbox가 없는 한 실행하지 않는다. 운영 사이트에서 로그인·입력·제출하는 E2E 테스트는 기본 테스트 집합에 포함하지 않는다.

## 구현 순서

1. 모델과 저장 계약: `ApplicantProfile`, `PostingRecord`, `EligibilityDecision`, `ApplicationTarget`, `ApplicationPackage`
2. 공식 도메인 탐색·본문 해시·변경 이벤트
3. 자격 판정 규칙과 `manual_review` 전환
4. 지원 패키지 생성 및 해시 검증
5. `review_required` Playwright 어댑터와 fixture 테스트
6. 사용자 승인 UI/CLI와 제출 증거 저장
7. 별도 설정·감사 로그·조건 검증 후 제한적 제출

각 단계는 Phase 1의 최종 manifest와 승인 근거 계약을 재사용하며, 자동 제출은 마지막 단계의 opt-in으로 남긴다.
